## Design Decisions

- **Project routing stays UUID-based** per spec 07 ("project ID and room ID stay aligned"). The `/editor/[roomId]` route resolves projects by UUID; `POST /api/projects` returns the UUID and the client navigates to `/editor/<uuid>`. The slug shown in the Create dialog preview is **display-only** and is not persisted or used for routing.
- **Sidebar shows a unique derived slug, not the UUID.** `ProjectSidebar` renders `projectSlug(project.name, project.id)` — the slugified name plus the first 6 hex chars of the project UUID (e.g. `payments-service-a3f2b1`). This distinguishes multiple projects with the same name. The real `project.id` is still used as the React key and for rename/delete callbacks. The Create dialog preview mirrors this format using a random 6-char suffix (the project ID doesn't exist yet at creation time).
- **Rename/delete buttons: hover-reveal on desktop, always visible on mobile.** Root cause of the "missing buttons" report: the action container used `opacity-0 group-hover:opacity-100` with no touch fallback, so on mobile/touch the buttons stayed at `opacity:0` permanently. Fix: `md:opacity-0 md:group-hover:opacity-100` — buttons are always visible below the `md` breakpoint and reveal on hover above it. The Tailwind v4 `group-hover` variant generates the expected `.group-hover\:opacity-100:is(:where(.group):hover *)` rule, so desktop hover works.
- **Create dialog label corrected** from "Room ID preview" to "Slug preview" to stop implying the slug is the URL/Liveblocks room ID. The `createSlug` is `slugify(createName)` + a random 6-char suffix, matching the `name-xxxxxx` format the sidebar will show once the project exists.

## Architecture Decisions

- Tailwind v4 CSS-first config — design tokens defined as CSS custom properties in globals.css, mapped to Tailwind via @theme inline.
- shadcn/ui components authored directly (no CLI); live in components/ui/.
- ProjectSidebar is a fixed overlay (does not push page content) positioned below the navbar.
- Auth uses Supabase Auth (not Clerk). Cookie-based sessions via @supabase/ssr. Middleware at project root handles session refresh and route protection. Public routes: /login, /signup, /forgot-password, /reset-password, /auth/callback.
- Editor page at /editor is a server component that fetches user then renders client EditorChrome with userEmail prop.
- Auth route pages (`app/(auth)/*/page.tsx`) are Server Components that render client form components. Avoid `"use client"` on `page.tsx` itself — Next.js wraps client pages in `ClientPageRoot`, which requires `workStore` for `searchParams` instrumentation and can throw in WebContainer environments (Bolt/StackBlitz).
- Database layer uses Bolt's integrated Supabase database (PostgreSQL) instead of Prisma. Schema is applied via the Supabase migration tool, not a Prisma schema file. All tables use RLS with owner-scoped policies.
- Artifact storage uses Supabase Storage instead of Vercel Blob. Canvas snapshots and generated specs are stored in Supabase Storage buckets, with the storage path stored as a reference column on the corresponding Supabase table.
- Background tasks continue to use Trigger.dev (not Supabase Edge Functions). Supabase Edge Functions have a 150s/400s wall clock limit and 2s CPU time limit, which is insufficient for durable multi-step AI generation workflows. Trigger.dev provides retries, realtime status streaming, and run-scoped tokens that would need to be rebuilt.

## Bug Fixes

- Fixed `Failed to delete project: TypeError: fetch failed` (recurring). The earlier 204→200 fix addressed the response body but the error persisted because the real root cause was at the transport layer: the DELETE `fetch()` sent no `Content-Type` header and no body, while the working POST/PATCH calls both send `application/json` with a JSON body. The Turbopack dev server drops bodyless DELETE requests to route handlers, so the project was never deleted server-side (it remained in the sidebar) and the client `fetch` threw `TypeError: fetch failed` — the undici wrapper error whose real cause (e.g. ECONNRESET) lives in `err.cause`, which the old catch block never inspected. Fix (three layers): (1) normalize the DELETE fetch to send `Content-Type: application/json` + an empty JSON body `{}`, matching the working POST/PATCH shape; (2) wrap the entire DELETE and PATCH route handler bodies in a single try/catch so any server-side throw returns a JSON 500 instead of crashing the function with no response; (3) add an `unwrapErrorMessage` helper that walks the `.cause` chain so the dialog and console show the real underlying error instead of the generic "fetch failed"; (4) add a direct-Supabase delete fallback in `submitDelete` — if the API route is unreachable, retry the delete via the browser Supabase client, which is safe because the `delete_own_projects` RLS policy (`auth.uid() = owner_id`) enforces ownership at the database level. Build passes.
- Fixed `cookies was called outside a request scope` error on `/editor`. Root cause: each query function in `lib/projects/queries.ts` called `createClient()` (which calls `cookies()`) independently, and when `app/editor/page.tsx` ran two queries via `Promise.all`, the concurrent branches lost Next.js's async request-store context (`workUnitAsyncStorage`), so `cookies()` threw `throwForMissingRequestStore`. Fix: call `cookies()` exactly once per request to create a single Supabase client, then pass that client into `getCurrentUser(supabase?)` and all query functions (`listOwnedProjects(supabase, ownerId)`, `createProject(supabase, params)`, etc.). Updated editor page and both API route handlers to use this shared-client pattern. Build passes; `/editor` remains dynamic (`ƒ`).
- Fixed `Failed to list shared projects: 'project' is not an embedded resource in this request` error on `/editor`. Two root causes: (1) the `project:...` embedded-resource join syntax in `listSharedProjects` was not recognized by PostgREST; (2) RLS policies on `projects` and `project_collaborators` only allowed project OWNERS to SELECT, so an invited collaborator could never read their shared projects. Fix: rewrote `listSharedProjects` to fetch matching `project_id`s from `project_collaborators` then fetch those project rows via `.in("id", projectIds)` (two simple queries instead of an embedded join); broadened both SELECT policies to allow owner OR collaborator (matched by `auth.jwt() ->> 'email'`) while keeping mutations owner-only. Applied via migration `fix_shared_projects_rls_and_access`. Build passes.
- Fixed `infinite recursion detected in policy for relation "projects"` error. Root cause: the broadened SELECT policies from the previous fix created a mutual recursion — `projects` SELECT subqueried `project_collaborators`, and `project_collaborators` SELECT subqueried `projects`; both tables have RLS enabled, so evaluating either policy re-entered the other indefinitely. Fix: extracted the cross-table ownership/collaborator checks into two `SECURITY DEFINER` SQL functions (`is_project_owner(uuid)`, `is_project_collaborator(uuid)`) that execute as their owner and bypass RLS internally, breaking the cycle. All policies on both tables now call these functions instead of subquerying the other table directly. Functions are granted `EXECUTE` only to `authenticated`. Applied via migration `fix_rls_recursion_with_security_definer_funcs`. Build passes; policies verified in `pg_policies`.
- Fixed case-insensitive email matching in collaborator RLS policies. The `is_project_collaborator` helper function and the `select_own_project_collaborators` SELECT policy compared `project_collaborators.email` against the JWT email claim with strict case-sensitive equality. Since Supabase Auth lowercases emails in `auth.users` and the JWT, a collaborator row stored with mixed-case casing (e.g. `User@Example.com`) would never match and the invited collaborator was silently denied access. Fix: applied `lower()` to both sides of the comparison in both locations via migration `fix_case_insensitive_email_in_collaborator_rls`. Access semantics (owner OR collaborator-by-email) preserved; mutation policies unchanged. Build passes.
- Sanitized Supabase error messages in project API responses. Previously each query function in `lib/projects/queries.ts` threw `` `Failed to <action>: ${error.message}` `` and the route handlers returned `err.message` verbatim in 500 responses, leaking raw PostgREST/RLS diagnostics (constraint/column names) to clients. Fix: introduced a `ProjectQueryError` class carrying a stable client-safe `message`, an `operation` tag, and the raw `detail`; query functions now throw `ProjectQueryError` with the safe message and route handlers use a shared `errorResponse` helper that logs the raw `detail` server-side (`console.error`) while returning only the client-safe `message` in the JSON body. Applied to all seven query functions and both route handlers (`/api/projects`, `/api/projects/[projectId]`). Build passes.
- Removed the direct-Supabase delete fallback from `submitDelete` in `hooks/use-project-actions.ts`. The fallback retried via the browser Supabase client on any non-2xx API response, which bypassed the DELETE route's 403/404 handling and could close the dialog even when RLS deleted nothing. The API route (`/api/projects/[projectId]` DELETE) now enforces auth + ownership and returns proper 401/403/404 status, so it is the only delete path; errors surface in the dialog via `unwrapErrorMessage`. Removed the now-unused `createClient` import. Build passes.
- Fixed `SECURITY DEFINER` function exposure on `is_project_owner` / `is_project_collaborator`. A security scan reported that `anon` and `authenticated` could execute both `SECURITY DEFINER` helper functions via PostgREST RPC endpoints (`/rest/v1/rpc/...`), letting any caller probe project ownership/collaborator status with the function owner's RLS-bypassing privileges. The previous `REVOKE FROM PUBLIC` did not remove Supabase's role-specific default `EXECUTE` grants to `anon`/`authenticated`, and revoking `authenticated` broke the RLS policies that call the functions (confirmed: `permission denied for function is_project_collaborator`). Fix (migration `fix_security_definer_function_exposure`): (1) converted both functions to `SECURITY INVOKER` so they execute under the caller's privileges and RLS — direct REST calls are now harmless; (2) rewrote the `project_collaborators` SELECT policy to be self-access only (`lower(email) = lower(auth.jwt() ->> 'email')`), removing the `is_project_owner` call that created the original mutual recursion with `projects` SELECT — owners will read collaborators via server-side service role per spec 09, while collaborators still see their own rows; (3) kept `projects` SELECT calling `is_project_collaborator` and `project_collaborators` mutations calling `is_project_owner` — under `SECURITY INVOKER` these re-enter RLS but no longer recurse (the `projects` SELECT short-circuits at `auth.uid() = owner_id` before reaching `is_project_collaborator`, and `project_collaborators` SELECT no longer references `projects`); (4) revoked `EXECUTE` from `anon` and kept the grant to `authenticated` only. Verified: both functions are `SECURITY INVOKER` with grants to `authenticated` + `service_role` only; `anon` excluded. Build passes.

- Fixed `Failed to delete project` error. Root cause: migration `fix_security_definer_function_exposure` converted `is_project_owner` from `SECURITY DEFINER` to `SECURITY INVOKER` to close a REST RPC exposure. This broke the CASCADE delete path: when `DELETE FROM projects` fires the `ON DELETE CASCADE` foreign key on `project_collaborators`, the `delete_own_project_collaborators` RLS policy calls `is_project_owner(project_id)`, which (as SECURITY INVOKER) queries `projects` under the caller's RLS — but the parent row is already gone, so the function returns FALSE, the policy blocks the cascade, and the foreign key constraint violation fails the entire delete. Fix: delete collaborators explicitly before the project in the DELETE route handler (`app/api/projects/[projectId]/route.ts`) via a new `deleteProjectCollaborators` helper in `lib/projects/queries.ts`. With child rows removed first, the CASCADE finds nothing to delete and the RLS policy is never evaluated during cascade. The `SECURITY INVOKER` security improvement is preserved. Build passes.

- Fixed `Failed to delete project collaborators` error. Root cause: the `project_collaborators` SELECT RLS policy (migration `fix_security_definer_function_exposure`) is self-row-only (`lower(email) = lower(auth.jwt() ->> 'email')`), so the project owner — who is NOT a collaborator on their own project — sees zero rows in the table. PostgREST implements DELETE by first SELECTing matching rows (subject to SELECT RLS), so `deleteProjectCollaborators` (which issued `.delete().eq("project_id", ...)`) found nothing and the operation failed. The same issue also broke `removeCollaborator` in `lib/projects/collaborators.ts` (`.delete().select("id")` returned an empty array, causing a false "Collaborator not found" error). Fix: two new `SECURITY DEFINER` Postgres functions that bypass RLS under owner verification — `delete_project_collaborators(project_uuid uuid)` deletes all collaborators for a project, and `remove_project_collaborator(project_uuid uuid, collaborator_email text)` deletes a single collaborator by email (case-insensitive, returns row count). Both verify ownership via `auth.uid()` directly and raise an exception for non-owners. `deleteProjectCollaborators` and `removeCollaborator` now call these via `supabase.rpc()`. This mirrors the established `get_project_collaborators` pattern. Applied via migrations `add_delete_project_collaborators_function` and `add_remove_project_collaborator_function`. Build passes.
